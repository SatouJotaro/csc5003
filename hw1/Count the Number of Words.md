# Count the Number of Words/统计单词数
> https://www.luogu.com.cn/problem/P1308
## 理解题意
这道题目源自 **NOIP2011 普及组第二题**，其核心要求是在给定的文章中统计某个单词出现的**次数**和**第一次出现的位置**。

为了让你更好地理解，我从题目含义、逻辑细节、输入输出三个方面为你详细拆解：

---

### 1. 题目含义（核心规则）

这道题要求实现一个类似文本编辑器中“单词搜索”的功能，必须遵守两个重要准则：

*   **大小写不敏感 (Case-insensitive)**：比如单词是 `To`，那么文章中的 `to`、`TO`、`tO` 都算作匹配成功。
*   **完全匹配 (Exact match)**：这是最容易“踩坑”的地方。并不是只要文章中有这个字母组合就算。
    *   **独立单词**：单词必须是被空格隔开的，或者位于文章的开头/结尾。
    *   **非子串**：如果文章中出现了单词的局部，则不能算数。
    *   *例子*：单词是 `to`。文章是 `Ottoman`。虽然 `Ottoman` 里包含了 `to`，但因为它是 `Ottoman` 这个单词的一部分，而不是独立单词，所以这里**不算匹配**。

---

### 2. 输入输出解释

#### 输入格式
*   **第 1 行**：一个单词（字符串，只含字母），长度不超过 10。
*   **第 2 行**：文章（字符串，含字母和空格），长度可达 $10^6$。

#### 输出格式
*   **找到时**：输出两个整数，用空格隔开。第一个是“出现总次数”，第二个是“第一次出现的位置”（位置从 0 开始计数）。
*   **找不到时**：直接输出 `-1`。

---

### 3. 实例讲解

#### 样例 1：
*   **输入**：
    ```
    To
    to be or not to be is a question
    ```
*   **逻辑**：
    *   文章中的第一个 `to` 在索引 0 位置，它前面是开头，后面是空格，完全匹配。
    *   文章中的第二个 `to` 在索引 13 位置，它前后都有空格，完全匹配。
*   **输出**：`2 0` （出现了2次，第一次出现在第0位）。

#### 样例 2：
*   **输入**：
    ```
    to
    Did the Ottoman Empire lose its power at that time
    ```
*   **逻辑**：
    *   尽管 `Ottoman` 中包含了 `to`，但它不是独立的单词。
    *   尽管 `at` 的末尾有 `t`，但整个单词不是 `to`。
    *   文章中没有独立的 `to`。
*   **输出**：`-1`。

---

### 4. 编程算法思路提示（如何处理 $10^6$ 规模的数据）

如果你用编程实现，建议注意以下步骤：

1.  **转换统一格式**：为了处理大小写不敏感，可以将单词和文章全部转为小写。
2.  **定位技巧**：
    *   不要用简单的 `str.find()`，因为那会匹配到 `Ottoman` 中的 `to`。
    *   **方案一**：可以在单词的首尾两端加上一个空格手动处理（例如搜索 `" to "`），但要注意处理文章开头和结尾的情况。
    *   **方案二**：直接遍历文章，检查每一个位置 `i`：
        *   确定 `i` 是单词的起始位置（满足：`i` 为 0 或者 `article[i-1]` 是空格）。
        *   确定 `i+单词长度` 是单词的结束位置（满足：`i+长度` 等于文章长度 或者 `article[i+长度]` 是空格）。
        *   比较这部分内容是否与目标单词一致。
3.  **效率优化**：由于文章长度可达 $10^6$，尽量使用单次遍历查找，避免重复处理。

这道题考察的是字符串处理的严谨性以及对边界条件（开头、结尾、空格分隔）的把控，如果理解了“独立单词”的含义，实现起来并不复杂。

## 知识点
这道题目 **NOIP2011 普及组《统计单词数》** 是一道非常经典的**字符串处理（String Processing）**题目。它之所以被放在普及组第二题，是因为它并不依赖复杂的高级算法（如树、图或动态规划），而是极其考验**逻辑严谨性、对边界条件的控制，以及字符串处理的基础算法实现能力**。

以下是该题目涉及的核心知识点：

### 1. 数据结构 (Data Structures)
这道题不需要复杂的数据结构，主要涉及：
*   **字符串 (String / Char Array)**：这是最核心的存储结构。在 C++ 中，可以使用 `std::string` 或直接使用 `char[]` 数组。鉴于题目给出的数据规模是 $10^6$ 级别，合理使用 `std::string` 配合 `getline` 读取整行即可。
*   **数组 (Array)**：如果你倾向于底层处理，可以使用字符数组来模拟内存缓冲区，这在处理超长字符串时能够比较直观地观察到下标。

### 2. 算法与逻辑方法 (Algorithms)
虽然不需要复杂的算法（如 KMP 或 AC 自动机），但需要用到以下基础逻辑：

*   **线性扫描 (Linear Scanning)**：
    *   题目文章长度高达 $10^6$，这意味着算法的时间复杂度必须控制在 **$O(N)$**。即：你只能遍历一遍或常数遍文章，不能在每次匹配时都重新扫描一遍全文。
*   **滑动窗口 / 指针匹配 (Sliding Window / Pointer Matching)**：
    *   你需要维护一个类似“窗口”的机制，通过检查单词的起始和结束点是否符合“完全匹配”的条件（左右两边是否为空格或边界），来判断当前位置是否为一个合法的单词。
*   **标准化处理 (Normalization)**：
    *   为了处理“大小写不敏感”，你需要将输入的单词和文章中的单词统一转换成小写（或大写）。常用技巧是利用 ASCII 码：`'A'` 到 `'Z'` 的值通过 `+32` 变为 `'a'` 到 `'z'`。

### 3. 程序设计中的关键难点（容易失分点）
这道题真正考察的是你对**边界判定**的能力：

1.  **单词边界判断 (Boundary Condition Check)**：
    *   这是本题最大的难点。如何判断一个匹配是“完全匹配”？
    *   核心逻辑：如果文章中的子串匹配成功，必须检查：
        1.  该子串前面的字符是空格 或者 该子串正好位于字符串开头。
        2.  该子串后面的字符是空格 或者 该子串正好位于字符串结尾。
    *   *如果忽略这一点，就会把 `Ottoman` 里的 `to` 也算进去，导致错误。*

2.  **输入处理 (Input Handling)**：
    *   `cin >> string` 遇到空格会停止，这道题的第 2 行包含空格，因此必须使用 `getline(cin, s)` 来读取一整行。

3.  **下标越界保护**：
    *   在检查 `article[i-1]` 或 `article[i+len]` 时，务必先判断索引是否合法（是否小于 0 或 超过字符串长度），这是新手最容易出现 RE（运行时错误）的地方。

### 4. 总结与建议
如果你要解决这道题，掌握以下几个逻辑思维比掌握复杂算法更重要：

1.  **“归一化”思维**：先把一切变为统一格式（统一转小写），这样后续判断无需考虑大小写。
2.  **“定位”思维**：不要强迫单词必须被空格包裹，要分情况讨论（比如：开头如何？结尾如何？中间如何？）。
3.  **性能思维**：不要在匹配循环里写过多复杂操作，保持 $O(N)$ 的线性结构。

**总结：这不是一道考察“你会什么算法”的题，而是一道考察“你写的程序是否经得起边界测试”的题。** 只要逻辑缜密，配合简单的遍历即可通过。

## 代码实现细节
在ACM模式下（使用 `cin` 和 `cout` 进行标准输入输出），解决这道题的关键在于**如何高效且准确地处理字符串查找**。

你可以利用 C++ 标准库中的 `std::string` 提供的 `find` 方法来跳跃式地查找，并手动增加判断逻辑以确保“单词边界”的正确性。

### C++ 解题代码实现

```cpp
#include <iostream>
#include <string>
#include <algorithm>
#include <cctype>

using namespace std;

// 将字符串转换为小写，方便统一比较
void toLower(string &s) {
    for (char &c : s) {
        c = tolower(c);
    }
}

int main() {
    // 优化输入输出速度
    ios::sync_with_stdio(false);
    cin.tie(NULL);

    string word, article;
    
    // 读取单词（不含空格）
    cin >> word;
    // 吃掉单词后面的换行符
    string dummy;
    getline(cin, dummy); 
    // 读取整篇文章（含空格）
    getline(cin, article);

    // 转换为小写
    toLower(word);
    toLower(article);

    int count = 0;
    int first_pos = -1;
    int word_len = word.length();
    int article_len = article.length();

    // 在文章中循环查找单词
    size_t pos = article.find(word, 0);
    while (pos != string::npos) {
        
        // 检查边界条件
        // 1. 左边界条件：pos 是开头 或者 单词前一个字符是空格
        bool left_ok = (pos == 0) || (article[pos - 1] == ' ');
        
        // 2. 右边界条件：pos+长度 是文章末尾 或者 单词后一个字符是空格
        bool right_ok = (pos + word_len == article_len) || (article[pos + word_len] == ' ');

        if (left_ok && right_ok) {
            count++;
            if (first_pos == -1) {
                first_pos = (int)pos;
            }
        }

        // 继续往下寻找下一个匹配点
        pos = article.find(word, pos + 1);
    }

    // 输出结果
    if (count == 0) {
        cout << -1 << endl;
    } else {
        cout << count << " " << first_pos << endl;
    }

    return 0;
}
```

### 代码逻辑详解：

1.  **输入处理**：
    *   `cin >> word`：读取第一行的单词。
    *   `getline`：由于文章包含空格，必须使用 `getline` 读取整行。注意在读取 `word` 后有一个换行符残留，所以需要用一个临时变量 `dummy` 把换行符“吃掉”。
2.  **统一大小写 (`toLower`)**：
    *   将 `word` 和 `article` 全部转为小写，这是解决“大小写不敏感”最简单的方法。
3.  **核心查找逻辑 (`article.find`)**：
    *   我们使用 `string::find(word, pos)`，它能快速定位到字符串中某个子串出现的位置，比遍历逐个字符检查要快得多。
4.  **边界检查 (最重要的步骤)**：
    *   即便 `find` 找到了 `to`，如果不满足左右空格的条件，它是无效的。
    *   `pos == 0` 处理了单词位于文章开头的情况。
    *   `pos + word_len == article_len` 处理了单词位于文章末尾的情况。
    *   其余情况则检查 `article[pos-1]` 和 `article[pos + word_len]` 是否确实为空格。
5.  **性能**：
    *   `string::find` 在底层实现中通常是非常优化的（即使是朴素算法，其平均效率也非常接近 $O(N)$），对于 $10^6$ 的长度，此程序可以在限制时间内运行完毕。

### 为什么这样做是安全的？
这个实现方式避开了复杂的字符串匹配算法（如 KMP），因为它利用了 C++ `std::string` 内部的高效实现。只要在 `find` 之后补上严谨的逻辑判断，就完全能够满足 NOIP 题目的判定需求。

太棒了！这些问题都切中了 C++ 字符串处理和底层机制的命门，对于 ACM 模式编程来说，搞懂这些能让你少走很多弯路。

我们一个一个拆解：

---

### 1. 变量名旁的 `&` 是什么意思？
在 C++ 中，`&` 符号出现的地方不同，含义完全不同：

*   **作为函数参数 (类型后面带着 `&`)：例如 `void toLower(string &s)`**
    *   这叫 **引用 (Reference)**。
    *   **含义**：它相当于给原来的字符串 `s` 起了一个“别名”。函数内部操作的 `s` 就是函数外部传入的那个变量本身，**而不是它的拷贝**。
    *   **为什么用它？** 字符串可能很长（$10^6$ 个字符），如果直接写 `void toLower(string s)`，系统会把整个字符串复制一遍传入函数，非常浪费内存和时间。加上 `&` 不会产生拷贝，效率更高。
*   **作为取地址运算符：例如 `&a`**
    *   放在变量前面时，表示获取该变量在内存中的 **地址 (Address-of)**。
*   **总结**：在参数列表里出现 `类型 & 变量名` 就是引用；出现在普通表达式里就是取地址。

---

### 2. 输入处理与 `getline` 的陷阱
这是新手最容易掉进去的坑！

*   **`cin >> word` 有什么特点？** 它读入时会**忽略前面的空格**，并且以**空格或回车**作为结束标识，且它**不会读走回车符**。
*   **为什么要 `getline(cin, dummy)`？**
    *   当你执行 `cin >> word` 后，光标停在单词末尾，后面紧跟着一个**回车符（换行）**。
    *   如果你紧接着写 `getline(cin, article)`，`getline` 的机制是：**它从当前光标开始读，直到遇到换行符为止**。
    *   因为刚才输入 `word` 剩下的那个换行符在那儿，`getline` 会直接读取到这个换行符，于是立刻停止，导致 `article` 变成了一个空串！
    *   所以我们要用一个 `dummy`（哑变量）变量把那个多余的换行符“吃掉”。
*   **`getline(cin, s)` 是什么？**
    *   第一个参数 `cin`：表示从标准输入读入。
    *   第二个参数 `s`：表示读入的内容存入目标字符串。
    *   它能读入**包含空格**的整行内容，直到遇到换行符为止。

---

### 3. `tolower` 和为什么要写函数？
*   **C++ 有 `tolower` 吗？** 有，在 `<cctype>` 头文件里。
*   **注意**：标准库的 `tolower` 是专门处理**单个字符** `char` 的，它不能直接传入一个 `string`。
*   **为什么要写 `toLower` 函数？** 因为我们需要把一整段文章变成小写，所以需要遍历字符串，对每一个字符调用 `tolower`。写成函数只是为了代码结构清晰、逻辑复用。

---

### 4. `size_t` 是什么类型？
*   它是 `unsigned int`（或 `unsigned long`）的一个别名，意思是 **"Size Type"**。
*   在计算机中，它专门用来表示**内存大小、字符串长度或数组下标**。因为它永远不会是负数，所以如果你发现某个函数返回值是 `size_t`，说明它返回的肯定是一个正数或者 0。

---

### 5. `string::find` 和 `pos`
*   **`find` 的作用**：查找目标子串在字符串中第一次出现的位置。
*   **第一个参数**：你要找的单词。
*   **第二个参数**：代表从字符串的哪个位置开始寻找。设为 `0` 说明从头开始搜。
*   **`pos != string::npos` 的含义**：
    *   `npos` 是一个特殊的常量，代表 **"-1"**（或者说是该类型能表示的最大值）。
    *   如果 `find` 函数**没找到**单词，它就会返回 `string::npos`。
    *   所以这个逻辑的意思是：**只要还没搜到头（只要还能找到），就一直循环下去**。

---

这是一个非常专业的问题！在 ACM 模式中，代码的“简洁性”和“可读性”确实很重要。

我直接回答你的疑惑：**即使你用了 `using namespace std;`，你也必须写 `string::npos`，不能直接写 `npos`。**

#### 1. 为什么 `using namespace std;` 没把 `npos` 给“省”掉？
这是 C++ 的语法规则：`npos` 不是一个独立的全局变量，而是 `string` 这个类（类型）内部的一个**静态常量成员**。

*   `std::string` 是一个类。
*   `npos` 就像是这个类的“属性”。
*   所以，哪怕你把整个 `std` 的命名空间都拉进来了，你依然需要通过 `string::` 来指明它是属于 `string` 类的。这就像你可以直接用 `vector`，但访问 `vector` 的成员时依然要用 `vector::` 一样。

---

#### 2. ACM 模式下“更好看”的写法（IDIOM）
在 ACM 竞赛代码中，大家通常不把 `find` 和判断分开写两行，而是习惯把赋值和判断放在一个 `while` 循环里，这被认为是**最简洁、最标准的写法**：

```cpp
while ((pos = article.find(word, pos + 1)) != string::npos) {
    // 逻辑部分...
}
```

**这种写法的好处：**
1.  **代码更紧凑**：不需要在循环外写一次 `find`，循环内再写一次 `find`。
2.  **减少冗余**：避免了 `pos = ...` 这一行代码在两处重复出现，修改逻辑时不容易漏。

---

#### 3. 解密 `string::npos`
其实 `npos` 是 "No Position" 的简写。

为什么要用这么复杂的写法？
因为 `string::find` 返回的是 `size_t` 类型（本质是无符号整数）。如果没找到，它需要返回一个“不可能存在的位置”。
C++ 选择了 `unsigned` 类型所能表达的**最大值**（例如在 64 位系统下是 $2^{64}-1$）。

*   当你写 `!= string::npos` 时，它是通过数值比对来判断是否查找结束。
*   **小技巧**：你也可以记住，`npos` 就是 `-1` 的二进制补码形式（因为无符号数溢出后会变成最大值）。所以有些人会写 `while((pos = ...) != -1)`，但**极度不推荐**这么写，因为这会触发编译器的警告，并且显得很不专业。

---

#### 4. 有没有更“激进”的简化？
如果你特别追求代码行数，或者觉得 `string::npos` 实在太碍眼，你可以使用 `using` 别名（在函数外定义）：

```cpp
using std::string;
const auto NOT_FOUND = string::npos; // 别名定义

// 后面就可以这样写：
while ((pos = article.find(word, pos + 1)) != NOT_FOUND) {
    // ...
}
```

**我的建议：**
不要尝试去“精简”掉 `string::npos`。在 C++ 开发中，`... != string::npos` 就像是一句**“固定行话”**。只要你写出这段代码，任何一个 C++ 程序员（或者面试官）一眼就能看出你在处理字符串查找。**使用行业标准写法，既不容易出错，也显得你的代码功底扎实。**

#### 总结
*   `string::npos` 是语法强制要求的类作用域查找。
*   推荐使用 `while ((pos = s.find(...)) != string::npos)` 这种“赋值+条件”合一的写法，这是 C++ 的经典惯用写法（Idiom）。

继续保持这种追根究底的精神，ACM 模式代码写多了，你会发现这些“繁琐”的写法其实都是为了程序的健壮性设计的！还有别的部分困惑吗？

### 补充问答：为什么 `pos = article.find(word, pos + 1)` 这样写？

这里是整个匹配的核心：
1.  当我们查找到一个 `pos`（当前单词出现的位置）后，为了找下一个，我们必须从 `pos + 1` 的位置开始搜索。
2.  如果不加 1，你会无限循环卡在同一个位置找到同一个单词。
3.  **注意**：在正式的匹配算法中，通常会跳过一个单词的长度来查找以提高匹配速度，但在处理“单词位置”问题时，为了防止单词重叠等复杂情况，按 `pos + 1` 一格格往后找是最保险的。

### 总结一下你的疑问链：
*   **&**：在参数里是传引用，别名，为了省内存。
*   **getline 为什么吃换行**：因为 `cin >>` 留下的换行符是 `getline` 的终止符，不去掉它后面的读入就会直接结束。
*   **size_t & npos**：全是 C++ 字符串库的标准写法，用来处理“还没找到”和“字符串长度”这种无符号场景。
