**一、题目概览**

- 有一个长度为 `n (1 ≤ n ≤ 10⁶)` 的整数序列 `A = a₁, a₂, … , aₙ`（每个 `aᵢ` 在 `[1, 10⁶]` 之间）。
- 接下来有 `q` 条查询，每条查询给出一个长度为 `m (1 ≤ m ≤ 10⁶)` 的序列 `B = b₁, b₂, … , bₘ`（`∑ m ≤ 10⁶`）。
- 对每条查询要判断 **B 是否是 A 的子序列**（即能否在 A 中删除若干（不一定连续）元素后得到 B）。
- 对每个查询输出  
  - `TAK` （是子序列）  
  - `NIE` （不是子序列）

---

**二、输入 / 输出格式（完整版）**

| 行号 | 内容 |
|------|------|
| 1    | `n` |
| 2    | `a₁ a₂ … aₙ`（`n` 个整数） |
| 3    | `q` |
| 接下来 `2·q` 行形成 `q` 条查询： |
| 4    | `m₁`（第一条查询的长度） |
| 5    | `b₁ b₂ … bₘ₁`（第一条查询的序列） |
| 6    | `m₂` |
| 7    | `b₁ b₂ … bₘ₂` |
| …    | … |
| 2·q+2 | `m_q` |
| 2·q+3 | `b₁ b₂ … bₘ_q` |

> **注意**：题目描述里经常出现空行（样例中出现了空行），实际评测数据里**不一定**有空行。读取时只需要用普通的 `cin >>` 逐个读取即可，空格、换行都会被自动跳过。

**输出**：  
每条查询占一行，输出 `TAK` 或 `NIE`（全部大写）。

---

**三、涉及的知识与技巧**

| 知识点 | 说明 |
|--------|------|
| **子序列** | “在原序列中保持相对顺序的若干元素”，与子串不同，子序列的元素不必相邻。 |
| **预处理 + 二分查找** | 对每个可能的值（`1 … 10⁶`）预先保存它在 `A` 中出现的所有下标，然后对查询的每个元素进行二分查找，得到最近的、且在当前匹配位置之后的出现位置。 |
| **贪心匹配** | 为了在子序列匹配中使用二分查找，只需每次都“尽可能往前”匹配（即取 **最小的**合法下标），这就是贪心策略的正确性来源。 |
| **时间/空间复杂度** | 预处理 `O(n)`，`q` 条查询的总时间为 `O(∑ m·log n)`，空间 `O(n + V)`（`V = 10⁶` 为值域上限）。 |
| **大输入** | 使用 `ios::sync_with_stdio(false); cin.tie(nullptr);` 关闭同步，提高读取速度。 |

---

**四、解题思路（核心步骤）**

1. **预处理 A**  
   - 建立一个数组 `pos[value]`（`value ∈ [1, 10⁶]`），其中每个 `pos[value]` 是一个 `vector<int>`，保存所有 **下标**（0‑based）使得 `A[idx] == value`，按下标递增保存。  
   - 遍历一次 `A`，把 `i`（0‑based）压入 `pos[A[i]]` 中。  
   - 这一步只需要 `O(n)` 的时间和 `O(n)` 的额外空间（所有下标的总和就是 `n`）。

2. **处理每条查询**  
   - 读取 `m` 与查询序列 `b[0…m‑1]`。  
   - 设 `cur = -1` 表示已经匹配到的 `A` 中的最右位置（尚未匹配任何元素）。  
   - 对 `b` 的每个元素 `x`：  
     - 若 `pos[x]` 为空 → `x` 在 `A` 中根本没有出现，直接输出 `NIE`。  
     - 否则在 `pos[x]` 中使用 `std::lower_bound(pos[x].begin(), pos[x].end(), cur+1)` 查找 **第一个 > cur** 的下标。  
       - `lower_bound` 返回的迭代器指向的元素就是“离当前位置最近且仍在 A 之后的出现”。  
       - 如果该迭代器等于 `pos[x].end()` → 没有合法位置，输出 `NIE`。  
       - 否则把 `cur = *it`（即匹配成功），继续处理下一个 `b`。  
   - 若遍历完整个 `b` 都没有失败 → 输出 `TAK`。

3. **为什么贪心+二分是正确的？**  
   - 对于固定的 `B`，如果存在一种匹配方式把 `B` 作为子序列，那么 **一定**存在一种匹配方式使得每一步都取 **最小的可行位置**（即上述二分的结果）。  
   - 证明（简要）：  
     - 设当前已经匹配到 `A` 中的位置 `cur`，要匹配下一个元素 `x`。如果在 `cur` 之后有多个出现位置 `p₁ < p₂ < …`，取最小的 `p₁` 不会妨碍后续匹配，因为任何后面更大的位置 `pᵢ (i>1)` 都必然在 `p₁` 之后，使用 `p₁` 给后续元素留下更大的“剩余空间”。于是如果用 `p₁` 能成功匹配完整个 `B`，则用更大的位置 `pᵢ` 也一定能成功（只需把后续的下标整体向后平移），所以只要 **存在**一种成功匹配，就一定可以用最小的位置完成匹配。  
   - 因此，只要在任意一步找不到 `> cur` 的出现，就说明根本不存在任何子序列匹配，直接返回 `NIE`。

**五、复杂度分析**

| 项目 | 时间 | 空间 |
|------|------|------|
| 预处理 `A` | `O(n)` | `O(n)`（存放下标的 vector） |
| 单条查询（长度 `m`）| `O(m·log k)`，`k` 为该值在 `A` 中出现的次数，`log k ≤ log n` | `O(1)`（仅用几个变量） |
| 所有查询总和 `∑ m ≤ 10⁶` | `O(∑ m·log n) ≤ O(10⁶·log 10⁶) ≈ 2·10⁷` | — |
| 整体 | `O(n + ∑ m·log n)` | `O(n + V)`，其中 `V = 10⁶` 为值域上限（`pos` 数组本身占 `V+1` 个空 vector） |

在 `n、∑ m ≤ 10⁶` 的限制下，这个复杂度完全可以跑进 1 秒以内。

**六、参考实现（C++17）**

```cpp
#include <bits/stdc++.h>
using namespace std;

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);
    
    int n;
    if(!(cin >> n)) return 0;          // 防止空输入
    const int MAXV = 1'000'000;        // 题目给出的值域上限
    vector<vector<int>> pos(MAXV + 1); // pos[value] 保存所有下标（0-index）
    
    for (int i = 0; i < n; ++i) {
        int x; cin >> x;
        pos[x].push_back(i);
    }
    
    int q; cin >> q;
    while (q--) {
        int m; cin >> m;
        vector<int> b(m);
        for (int i = 0; i < m; ++i) cin >> b[i];
        
        int cur = -1; // 当前匹配到的 A 中的最右位置
        bool ok = true;
        for (int i = 0; i < m && ok; ++i) {
            int x = b[i];
            const vector<int> &vec = pos[x];
            if (vec.empty()) {
                ok = false;            // 根本不存在
                break;
            }
            // 在 vec 中寻找 > cur 的最小下标
            auto it = lower_bound(vec.begin(), vec.end(), cur + 1);
            if (it == vec.end()) {
                ok = false;            // 没有合适的出现位置
                break;
            }
            cur = *it;                 // 成功匹配
        }
        cout << (ok ? "TAK" : "NIE") << '\n';
    }
    return 0;
}
```

**代码要点说明**

1. **数组 `pos`**  
   - 大小 `MAXV+1`（即 `10⁶+1`），每个元素是一个 `vector<int>`。因为 `aᵢ` 与 `bⱼ` 都不超过 `10⁶`，可以直接用下标访问，避免使用 `unordered_map` 带来的常数开销。

2. **下标使用 0‑based**  
   - `cur` 初始为 `-1`，对应的“当前已经匹配的最右位置”。`lower_bound` 用 `cur+1` 作为查找下界，确保每一次匹配的元素在 `A` 中严格位于上一次匹配位置的右侧，从而保持子序列的顺序。

3. **时间/空间**  
   - `pos` 中的所有 `vector` 总共容纳 `n` 个整数，约 `4 MiB`（`int` 4 字节），满足内存限制。  
   - 采用 `lower_bound`（二分）来定位下一个出现位置，单次查询的复杂度是 `O(m·log k)`，其中 `k` 为该值在 `A` 中出现的次数，实际运行非常快。

4. **输入容错**  
   - `cin` 会自动跳过空行和多余空格，所以不必手动跳过样例中的空行。  
   - 关闭同步、解除 `cin.tie` 提升 I/O 性能。

**七、完整思考过程（如果想自己手写）**

1. **先把问题抽象**：判断 `B` 是否为 `A` 的子序列 → 对每个查询元素在 `A` 中寻找“后面最近的相同值”。  
2. **为什么必须预处理**？直接逐字符比较会导致 `O(n·m)`，在本题极限下不可接受。  
3. **预处理怎么设计**？把所有相同数字的出现位置记下来——这是一条 **“值→出现位置集合”** 的映射。  
4. **查询时怎么使用**？对每个查询元素使用二分查找得到最近的可用位置。  
5. **是否会出现匹配冲突**？只要每次取 **最小的可用位置**（贪心），后面的元素仍然拥有足够的“剩余空间”。如果最小的位置不行，则任何更大的位置也不行（因为更大的位置会把后面的可用空间进一步压缩）。  
6. **复杂度是否可接受**？预处理 `O(n)`，每次查询 `O(m·log n)`，总规模 `≤ 10⁶`，完全在时间/空间限制内。

**八、常见错误与调试技巧**

| 常见错误 | 表现 | 解决办法 |
|----------|------|----------|
| 使用 `cur = -1` 但 `lower_bound` 传入 `cur` 而不是 `cur+1` | 可能匹配到同一位置，破坏子序列的“顺序”。 | 确保查找下界为 `cur+1`（因为下标是严格的递增）。 |
| 直接遍历 `pos[x]` 并手动找 `>cur` | 在最坏情况下会退化为 `O(k)`，导致总时间超限。 | 使用 `std::lower_bound`（二分）保证 `O(log k)`。 |
| 忘记 `vector<int> pos[MAXV+1]` 会产生巨大的空向量（`10⁶+1` 个），导致编译时的栈溢出。 | 编译或运行错误。 | 用 `vector<vector<int>> pos(MAXV+1);` 在堆上分配。 |
| 输入中出现了额外的空行，导致读取错误 | `cin` 仍然正确，但若手动用 `getline` 可能卡住。 | 使用 `cin >>` 逐个读取，空格、换行都会被自动忽略。 |
| 未关闭同步导致 I/O 速度慢，测评超时。 | 程序在读取/写入时卡顿。 | `ios::sync_with_stdio(false); cin.tie(nullptr);` 放在 `main` 开始处。 |

**九、总结**

- **核心**：对每个可能的数字预先记录所有在 `A` 中出现的位置 → 形成“值→位置集合”。  
- **查询**：从左到右依次匹配 `B`，每次用二分查找在对应集合里找 **大于当前匹配位置** 的最小下标。  
- **正确性**：若任意一步找不到合法位置，则 `B` 一定不是子序列；若所有步骤都成功，则得到的匹配方式保证是子序列。  
- **复杂度**：预处理 `O(n)`，整体查询 `O(∑ m·log n)`，空间 `O(n + 10⁶)`，完全满足题目约束。
